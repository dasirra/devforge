Explore the codebase first; answer from it instead of asking. Scope check, then walk the decision tree at product altitude: behavior, edge states, scope, success criteria. Technical unknowns get parked, not asked.

3/forge:building

One phase of issues per run: one worktree, one contract, one PR. No upfront technical plan; the contract is the plan. Evaluator never sees generator transcripts.

4

Execution

No technical plan, no PLAN.md: the contract is the plan. The lead splits criteria into workstreams with file territories inline; one sonnet teammate per workstream makes its own technical decisions against live code.

builders · sonnet × Nprogress.json
5

Static verification

Build, tests, lint, types. Real output only; the artifact must start before Phase 6. The builders' own tests run here, and here only. They are not evaluation.

6

Behavioral evaluation ⚔ · the core

A fresh evaluator exercises every criterion black-box through the surface: driving a browser, calling the public API, or running argv and reading exit codes. It has not read the implementation and does not run the builders' tests. Pass/fail per criterion, per issue; each round's fixer is fresh and sees the critique, never the evaluator's transcript.

Evaluation surfaces

The contract is verified by observing the running artifact from outside. What "outside" means depends on what the project ships, so Phase 0.5 resolves it before anything expensive happens. Only web needs a browser automation MCP server.

⚔ Adversarial pairs

  • planner vs critic · /forge:planning P3, PM plan quality
  • generator vs evaluator · /forge:building P2, what "done" means
  • evaluator vs fixer · /forge:building P6, running artifact vs contract

👤 Human touchpoints

  • /forge:interview · the alignment that can't be delegated
  • surface confirmation · one question in preflight
  • issue review · edit + arbitrate on GitHub before /forge:building
  • --gate · optional contract approval
  • PR review & merge · per phase, taste + QA

Separation rule

Critics and evaluators receive artifacts only, never the other agent's transcript, reasoning, self-assessment, implementation, or tests. Muddied context produces rubber-stamp evaluations, and a builder's green test suite certifies whatever misunderstanding produced the bug. Contracts change only via the amendment channel; never weakened to make a round pass.
